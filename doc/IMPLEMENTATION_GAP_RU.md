# Оценка готовности по 10 задачам (качественная версия)

Дата оценки: 2026-04-25.

## Сводно

- Общая готовность по 10 пунктам: **~58%**.
- Готовность **MVP (ваши 5 пунктов)**: **~76%**.

## 1) Импорт спектров и треков — **55%**

### Что уже есть
- Есть импорт **RadiaCode XML** и разбор базовых полей: время начала/конца, serial, имя устройства, длительность, спектр, калибровочные коэффициенты.
- XML-поток подключён в общий upload pipeline.

### Что осталось сделать
- Читать из XML (или сопутствующих данных), если доступны:
  - координаты;
  - высоту (и/или рассчитывать её);
  - погоду по координатам + времени + высоте;
  - dose rate как отдельное поле спектра (если присутствует в конкретном формате);
  - CPS как явное поле спектра (сейчас используется в трековых форматах, но не как часть модели спектра).
- Добавить единый объект `SpectrumContext` (coords/alt/weather/cps/dose/name/duration/source file).
- Добавить тесты на несколько вариантов XML (с/без доп. полей).

## 2) Привязка спектра к карте — **80%**

### Что уже есть
- Есть авто-поиск кандидатов трека по времени окна измерения и эвристике совпадения устройства.
- Есть ручной сценарий: выбрать точку или область на карте, если автоматическая привязка неуверенная.
- В UI показывается окно измерения и кандидаты.

### Что осталось сделать
- Добавить привязку к **конкретному сегменту маршрута** (start marker id / end marker id).
- Сохранять и показывать точную привязку «спектр ↔ marker/segment» в БД.
- В отчёте на карте явно показывать высоту в точке привязки.

## 3) Базовая нормализация — **35%**

### Что уже есть
- Энергия считается из calibration coefficients (keV).
- Есть базовый peak detection.

### Что осталось сделать
- Нормировать спектр в **counts/sec** на уровне модели/аналитики (а не только сырые каналы).
- Ввести стандартные **энергетические окна** (low/mid/high + прикладные ROI).
- Реализовать сравнение **«точка против локального фона рядом»** (по ближайшим точкам/соседнему времени).
- Хранить флаг, что нормализация выполнена (версия алгоритма).

## 4) Учёт детектора — **20%**

### Что уже есть
- В треках хранится detector/device metadata.
- При линковке используется имя/serial устройства.

### Что осталось сделать
- Добавить отдельный справочник/профиль детектора для спектральной части:
  - тип кристалла;
  - ориентировочное разрешение по энергии;
  - размер кристалла;
  - рабочий диапазон.
- Подстроить пороги поиска пиков и допуски линий под профиль детектора.
- Явно ограничить «смелость» выводов для low-resolution приборов.

## 5) Поиск значимых пиков — **50%**

### Что уже есть
- Есть локальный поиск пиков и ранжирование.
- Есть confidence на уровне сопоставления линий.

### Что осталось сделать
- Перейти от «локальный максимум > порог» к статистическому критерию значимости (SNR / Poisson background).
- Для каждого пика сохранить:
  - centroid energy;
  - ширину (FWHM/эквивалент);
  - площадь пика;
  - confidence как статистический, а не только по delta energy;
  - метку «полный пик / возможно комптоновский горб / шум».

## 6) База основных изотопов и групп — **65%**

### Что уже есть
- Есть большой единый каталог с ключевыми практичными изотопами (K, U/Ra, Th, Cs, Co, Am, I и т.д.).
- Есть групповые проверки по ключевым линиям (K/Cs/Co/U-Ra/Th/Am).

### Что осталось сделать
- Ввести «оперативный practical profile» как базовый режим (только приоритетные группы из задачи).
- Вынести экзотические/редкие нуклиды в расширенный режим, чтобы снизить ложные гипотезы.
- Добавить явную политику для Pu-239: «маловероятно без спецусловий».
- Добавить отдельный класс «unknown/unresolved» как first-class результат.

## 7) Проверка не одного пика, а группы признаков — **85%**

### Что уже есть
- Реализованы group rules с обязательными линиями и MinMatches.
- В отчёте есть confirmed / not confirmed по группам.

### Что осталось сделать
- Уточнить пороги по детектору и времени набора статистики.
- Добавить штраф за отсутствие ключевой сильной линии при «подтверждении» группы.
- Добавить устойчивую проверку против ложных совпадений на низких энергиях.

## 8) Простое разложение вклада — **70%**

### Что уже есть
- Есть coarse component estimation (K/U-Ra/Th/Cs/Co/Am и unknown floor).
- Выдаётся человеко-понятная интерпретация.

### Что осталось сделать
- Привести вывод к формату диапазонов вкладов (например 40–60%).
- Явно добавить low/mid/high energy contribution как отдельные строки.
- Вывести «не объяснено» как остаток после объяснённых групп с диагностикой причин.

## 9) Контекстная фильтрация — **5%**

### Что уже есть
- Есть место для контекста в UI-панели результата (через `spectrum_context`), но физические фильтры почти не применяются.

### Что осталось сделать
- Добавить слой контекстных правил (объект/локация/условия) как post-filter гипотез.
- Использовать тип местности/объекта и, при наличии, дополнительные признаки (photo tags / user label).
- В отчёте показывать: «какие варианты ослаблены/отброшены и почему».

## 10) Понятный отчёт на карте — **72%**

### Что уже есть
- Есть модальное окно с:
  - qualitative summary;
  - estimated components;
  - interpretation;
  - group confirmation;
  - measurement window;
  - действия для ручной привязки.

### Что осталось сделать
- Вынести это в стабильный «карточный» формат для точки/участка, не только upload-modal.
- Добавить раздел «что не подтверждено» в более явном и простом виде.
- Добавить короткое объяснение на простом языке на уровне каждой точки/сегмента.
- Добавить экспорт отчёта (JSON + человекочитаемый текст).

---

## Отдельно: готовность MVP из 5 пунктов

1. Загрузка спектра — **80%** (есть, но неполные поля контекста).
2. Привязка к карте — **80%** (есть авто + ручной сценарий, нет segment-level persistence).
3. Поиск пиков — **50%** (есть базовый, нет строгой статистики).
4. Проверка групп (K/U/Ra/Th/Cs/Co/Am...) — **85%** (в целом реализовано).
5. Понятный вывод «чем вызвано повышение» — **85%** (есть explanation/components/group checks).

Итоговый MVP: **~76%**.

## Рекомендуемая дорожная карта (порядок)

1. **Нормализация + статистика пиков** (пункты 3 и 5).
2. **Профили детекторов** (пункт 4).
3. **Сегментная привязка + хранение связей спектра** (пункт 2).
4. **Practical isotope profile + policy unknown/Pu** (пункт 6).
5. **Контекстная фильтрация + финальный формат отчёта** (пункты 9 и 10).

Это даст максимальный прирост качества при минимальном усложнении архитектуры.
